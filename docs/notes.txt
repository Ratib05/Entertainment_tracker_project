Security vulns:

Findings (prioritized)

- **Service role key usage — Risk: High**: The server creates a Supabase client using the service role key and that same client is used for most DB operations (privileged key in server code).
  - Evidence: supabase.service.ts
  - Why it matters: The service role key bypasses Row-Level Security (RLS). If any endpoint fails to enforce per-user filtering, it can allow privilege escalation or unauthorized data access.
  - Recommendation: Use the service role key only for truly privileged operations. For per-user queries either:
    - Enforce RLS on the DB and use an anon key or use the user's JWT with the Supabase client (set auth to the user JWT) so RLS applies; or
    - Maintain two clients (service client for privileged admin ops, and a user-scoped client or RPCs that enforce RLS) and audit every service method to ensure the correct client is used.
  - Files to review for enforcement: service creation supabase.service.ts, and service usages (examples): library.service.ts, lists.service.ts, entertainment.service.ts.

- **Missing/incorrect security headers — Risk: Medium**: The app sets `X-XSS-Protection` to `'0'`, and CSP/HSTS are not applied.
  - Evidence: main.ts
  - Why it matters: `X-XSS-Protection: 0` disables an older browser mitigation; absence of CSP and HSTS reduces defense-in-depth against XSS and mixed-content/transport downgrade attacks.
  - Recommendation: Remove the `X-XSS-Protection` header or set modern headers via `helmet()` (recommended). Add Content-Security-Policy and Strict-Transport-Security in production. Example: apply Nest/Express `helmet()` and tune CSP.

- **In-memory rate limiter — Risk: Medium**: Rate limiting is implemented with a static in-memory Map (single-process).
  - Evidence: rate-limit.guard.ts
  - Why it matters: In-memory limiters do not scale across cluster/replica instances and are bypassable by rotating IPs; they can be memory/CPU DoS targets.
  - Recommendation: Replace with a distributed store backed rate limiter (Redis-based) or a battle-tested middleware like `express-rate-limit` with a Redis store or `rate-limiter-flexible`. Emit and monitor rate-limit headers.

- **Error message / stack exposure in non-prod — Risk: Medium**: The global exception filter includes exception messages (and logs stack traces).
  - Evidence: http-exception.filter.ts
  - Why it matters: In non-production, returning detailed error messages can leak internal implementation details and sensitive information.
  - Recommendation: Return generic client messages; record stack traces in server logs only. Ensure `NODE_ENV` is set in production.

- **CORS configuration risks — Risk: Medium**: CORS is enabled if `CORS_ORIGIN` is set and the value is used directly; a wildcard or misconfigured origin could allow unwanted cross-origin requests.
  - Evidence: main.ts
  - Recommendation: Validate/whitelist origins (do not accept `*` if credentials used). Explicitly control allowed methods, headers, and credentials.

- **Missing HTTPS enforcement / HSTS — Risk: Medium**: No HSTS header or HTTPS redirect enforcement in code.
  - Evidence: no HSTS logic in main.ts
  - Recommendation: Enforce HTTPS at load balancer or app level and add HSTS in production.

- **Secrets management & accidental exposure — Risk: High (operational)**: Ensure SUPABASE_SERVICE_ROLE_KEY and other secrets are never committed, rotated regularly, and stored in a secret manager.
  - Evidence: service-role key referenced in code: supabase.service.ts; .env.example does not show service role key (good), but verify no commits contain secrets.
  - Recommendation: Use CI/CD secret stores, restrict who can access keys, rotate keys, and add a pre-commit hook / secret-scan to prevent accidental commits.

- **Public endpoints returning all fields — Risk: Low → Medium**: Public endpoints like entertainment GETs return `*` (all columns).
  - Evidence: entertainment.service.ts
  - Why it matters: If DB rows contain sensitive fields, `select('*')` could leak them.
  - Recommendation: Explicitly select only required public fields or sanitize responses.

- **Dependency & supply-chain risk — Risk: Medium**: Several dependencies should be audited and kept up to date (`@supabase/supabase-js`, `axios`, `passport`, Nest deps).
  - Evidence: package.json
  - Recommendation: Run `npm audit`, use lockfile, enable Dependabot or Snyk, and update critical dependencies promptly.

- **Inconsistent env var usage (src vs compiled dist) — Risk: Low**: Dist folder references might expect a different env var (e.g., ANON vs SERVICE key); verify runtime uses correct env names.
  - Evidence: compiled dist mentions `SUPABASE_ANON_KEY` while source uses `SUPABASE_SERVICE_ROLE_KEY`.
  - Recommendation: Standardize env var names and document them; ensure build/runtime use same vars.

  - `cd backend/nest-api && npm outdated`

  - then in bootstrap: `app.use(helmet());` and configure CSP for production.

Minimal prioritized remediation roadmap

2. Add/replace security headers via `helmet()` and add CSP/HSTS. (Medium)
3. Replace in-memory rate limiter with Redis-backed limiter or a cloud API gateway rate-limits. (Medium)
4. Harden error responses (do not return internal messages in non-prod incorrectly). (Medium)
5. Run `npm audit` and update high/critical deps; enable automated dependency scanning. (Medium)
6. Verify CORS policy and disallow wildcards/credentials unless necessary. (Medium)
7. Ensure secrets are in secret manager and rotate service keys. (High/operational)
